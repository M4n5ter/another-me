# Another Me记忆

## 工作流程设想:

1. **记忆摄入 (Ingestion)**:

- 新的外部信息（网页浏览、聊天、用户输入）或内部信息（Agent 思考）产生。
- 创建 MemoryUnit 记录，填充 type, source, content_text, metadata。
- 对 content_text 进行向量化，存入 content_embedding。
- 识别 content_text 中的实体，创建或查找 Entity 节点，并建立 MENTIONS 关系。
- 与上一条 MemoryUnit 建立 SEQUENCED 关系。
- 初步评估 importance。

2. **记忆检索 (Retrieval)**:

- 当 Agent 需要响应、决策或反思时，根据当前上下文生成检索查询。
- 混合查询:
  - 向量搜索: 在 MemoryUnit, Entity, UserPreference 上使用 content_embedding/representation_embedding 查找语义相似的记忆/偏好。
  - 图遍历: 从相关的 Entity 或最近的 MemoryUnit 出发，沿着 SEQUENCED, MENTIONS, SUPPORTS 等关系扩展，获取上下文相关的记忆。
  - 条件过滤: 使用 timestamp, type, source, importance 等元数据进行过滤。
- 排序与融合: 对检索结果根据相关性、重要性、时间近近性等进行排序，形成输入给 LLM 的上下文。

3. **记忆反思与整合 (Reflection & Consolidation - 定期或触发)**:

- Agent 检索一段时间内（或关于特定主题）的 MemoryUnit 集合。
- 使用 LLM 分析这些记忆，寻找模式、规律、用户反馈。
- 生成推断:
  - 如果发现一贯的行为模式或明确的用户表达，创建或更新 UserPreference 节点。
  - 建立 SUPPORTS 或 CONTRADICTS 关系连接证据 MemoryUnit。
  - 更新 UserPreference 的 confidence 和 last_updated。
- 生成摘要:
  - 将一组相关的低级 MemoryUnit 总结成一个 type='SUMMARY' 的 MemoryUnit。
  - 建立 ABSTRACTED_INTO 关系。
- 更新重要性: 根据访问频率、是否被用于推断、用户反馈等调整 MemoryUnit 的 importance。

## 记忆存储

可以不进行强制的长期记忆压缩或遗忘，而是依赖于高效的检索机制，实现“按需回想”。
具体来说，这种机制可以这样工作：

1. **核心上下文构建**: 当 another-me 需要进行思考、决策或响应时，首先构建一个包含最直接相关信息的核心上下文。这通常包括：

- 当前的触发事件/观察 (MemoryUnit - OBSERVATION, USER_FEEDBACK, COMMUNICATION 等)。
- 紧邻的短期记忆序列 (通过 SEQUENCED 关系或 timestamp 查询最近的 N 条 MemoryUnit)。
- 可能相关的、高置信度的核心 UserPreference。
- 当前的 AGENT_THOUGHT 或 GOAL（如果存在）。

2. **生成检索查询**: 基于这个核心上下文，生成一个或多个用于检索的查询。这可以简单地是当前观察/思考的文本内容的向量 (content_embedding)，也可以是 LLM 生成的更具体的查询语句（例如，“回想一下用户过去对[特定主题]的看法”）。

3. **动态记忆检索 (实现“回想”)**:

- 向量搜索: 利用核心上下文的嵌入或生成的查询嵌入，在 SurrealDB 的 MemoryUnit 表中执行向量搜索，查找 content_embedding 语义相似的历史记忆。可以过滤掉已经在核心上下文中的近期记忆。
- 图查询 (可选补充): 可以从核心上下文涉及的 Entity 出发，在图关系中查找相关的历史 MemoryUnit（例如，通过 MENTIONS 关系）。
- 元数据过滤: 可以结合 type, importance, source 等元数据进一步筛选检索结果。

4. **结果排序与选择**: 对检索到的历史记忆进行排序（基于相似度得分、importance、时间等因素），选择最相关的 Top-K 条记忆。

5. **上下文增强**: 将这些被“回想”起来的、高度相关的历史记忆补充到核心上下文中，形成最终提供给 LLM 的完整上下文。可以在 Prompt 中明确标注这些是“相关历史记忆”或“回想内容”，以引导 LLM 更好地利用它们。

6. **LLM 处理**: LLM 基于这个增强后的、包含近期信息和相关历史回想的上下文进行推理和生成。

**优势**:

- 无需信息丢失: 所有记录的 MemoryUnit 都完整保存在数据库中，避免了压缩或遗忘可能带来的信息损失。
- 上下文相关性: LLM 的上下文总是聚焦于当前任务最相关的记忆，而不是被大量无关或陈旧的信息淹没。
- 模拟人类记忆: 非常符合人类记忆的工作方式——我们并不会时刻意识到所有记忆，而是根据当前情境触发相关内容的“回想”。
- 可扩展性: 即使记忆库增长到非常庞大的规模，只要检索机制高效，LLM 的处理负担也不会无限增加。

**潜在考虑**:

- 检索效率: 随着数据量增长，需要确保持续优化 SurrealDB 的向量索引和查询性能。
- 检索质量: 需要不断调整检索策略（查询生成方式、Top-K 选择、排序逻辑），确保能准确、高效地“回想”到真正有用的信息，避免引入噪声。
- 存储成本: 虽然存储成本相对较低，但无限增长的数据量仍需考虑长期成本。不过，相比于压缩/遗忘可能丢失关键信息的风险，保留完整数据通常是值得的。

## 记忆重要性评分

可以设计一个多因素加权模型，结合多种启发式规则来计算 `importance` (0-1)。以下是一些关键因素和计算逻辑的设想：

**核心影响因素:**

1.  **用户交互强度 (User Interaction Intensity - `S_interaction`):** 用户直接参与或反馈的记忆通常更重要。
2.  **来源类型/可靠性 (Source Type/Reliability - `S_source`):** 某些来源（如用户指令）比其他来源（如后台日志）更重要。
3.  **情感显著性 (Emotional Salience - `S_emotion`):** 带有强烈情感色彩的记忆（无论正负）往往更关键。
4.  **与核心模型的关联度 (Relation to Core Model - `S_relation`):** 与已知用户偏好、习惯或目标的关联程度。
5.  **未来效用/被引用次数 (Future Utility/Retrieval Count - `S_utility`):** 被后续思考或行动频繁检索和使用的记忆，其重要性会提升。
6.  **信息新颖度/惊喜度 (Information Novelty/Surprise - `S_novelty`):** 与已知模式显著不同的记忆可能标志着变化或学习机会。
7.  **时间衰减 (Time Decay - `S_recency`):** 虽然不是绝对的（旧的基础记忆可能仍然很重要），但近期记忆通常对当前状态影响更大。
8.  **抽象层级 (Abstraction Level - `S_abstraction`):** 经过反思总结的记忆可能比原始观察更具概括性重要性，但原始观察是证据基础。

**计算逻辑设计:**

我们可以为每个 `MemoryUnit` 维护一个基础重要性 (`base_importance`) 和一个动态调整因子 (`dynamic_modifier`)，最终 `importance` 是两者的结合，并限制在 [0, 1] 区间。

**1. 基础重要性 (`base_importance`) - 创建时计算:**

这部分基于记忆单元创建时的固有属性。可以使用加权平均：

`base_importance = w1*S_interaction + w2*S_source + w3*S_emotion + w4*S_relation + w5*S_novelty + w6*S_abstraction`

(权重 `w1` 到 `w6` 相加为 1，需要根据实验调整)

-   **`S_interaction` (0-1):**
    -   直接用户指令/命令: 1.0
    -   明确的用户反馈 (确认/修正): 0.9
    -   用户主动发起的对话: 0.8
    -   用户对 Agent 建议的采纳/执行: 0.7
    -   无直接用户交互 (如观察): 0.3 - 0.5 (根据来源调整)
-   **`S_source` (0-1):**
    -   `USER_INPUT`, `USER_FEEDBACK`: 1.0
    -   `CHAT_API`: 0.9
    -   `AGENT_REFLECTION` (生成 `UserPreference` 或 `SUMMARY` 时): 0.8
    -   `BROWSER_PLUGIN` (主动浏览): 0.6
    -   `SCHEDULER` (如新闻推送): 0.5
    -   后台观察/系统日志: 0.3
-   **`S_emotion` (0-1):**
    -   使用情感分析工具 (如 VADER 或 LLM 判断) 评估 `content_text` 的情感强度 (取绝对值，归一化到 0-1)。
    -   无明显情感: 0.5 (基线)
-   **`S_relation` (0-1):**
    *   创建时直接关联到高置信度 `UserPreference` / `Goal`: 1.0
    -   关联到中置信度: 0.7
    -   关联到低置信度: 0.4
    -   未直接关联: 0.2
-   **`S_novelty` (0-1):**
    -   计算 `content_embedding` 与近期/相关记忆库中最相似记忆的距离 (1 - max_similarity)。如果距离很大 (低相似度)，则新颖度高。需要设定阈值或归一化。
    -   默认值: 0.5
-   **`S_abstraction` (0-1):**
    -   `INFERRED_PREFERENCE`, `SUMMARY`, `GOAL`: 0.9
    -   `AGENT_THOUGHT`: 0.7
    -   原始 `OBSERVATION`, `ACTION`, `COMMUNICATION`: 0.5

**2. 动态调整因子 (`dynamic_modifier`) - 随时间变化:**

这个因子反映记忆随时间推移和使用情况的变化。可以使用乘法或加法调整，这里用乘法因子（基数为 1）举例：

`dynamic_modifier = (1 + S_utility_boost) * S_recency_decay`

-   **`S_utility_boost` (>= 0):**
    -   每次该 `MemoryUnit` 被成功检索并用于推理/决策时，小幅增加一个“效用分数” (e.g., `utility_score += 0.1`)。
    -   `S_utility_boost = tanh(utility_score / k)` (使用 tanh 使增长饱和，k 是调节饱和速度的参数，e.g., k=5)。
-   **`S_recency_decay` (0-1):**
    -   `S_recency_decay = exp(-lambda * time_elapsed_in_days)`
    -   `lambda` 是衰减率，例如 `0.005` (非常缓慢的衰减)。
    -   **注意:** 对于某些类型的记忆（如已确立的核心 `UserPreference` 的证据 `MemoryUnit`），可以设置一个最低衰减下限，甚至不衰减 (`lambda=0`)。

**3. 最终重要性计算:**

`importance = clamp(base_importance * dynamic_modifier, 0, 1)`

或者可以采用更复杂的融合方式，例如让 `base_importance` 提供基础值，`dynamic_modifier` 在其上下浮动。

**计算时机:**

-   `base_importance` 在 `MemoryUnit` 创建时计算一次。
-   `dynamic_modifier`：
    -   `S_utility_boost` 在每次成功检索后更新。
    -   `S_recency_decay` 可以在每次检索时实时计算，或者定期 (如每天) 通过后台任务批量更新所有记忆单元的最终 `importance` 值。定期更新可能更高效。

**关键考虑:**

-   **权重调整:** 上述权重 (`w1` 到 `w6`) 和参数 (`k`, `lambda`) 需要通过实验和观察 Agent 的实际行为来仔细调整。
-   **计算成本:** 情感分析、新颖度计算可能需要调用 LLM 或其他模型，需要在性能和成本之间取得平衡。`S_novelty` 可以简化或仅在反思阶段计算。
-   **初始值:** 新创建的记忆单元需要一个合理的初始 `importance`。
-   **重要性传播:** 可以考虑在图关系中传播重要性，例如，一个非常重要的 `MemoryUnit` 可以略微提升其相邻（`SEQUENCED`）或相关（`MENTIONS`）记忆单元的重要性。

这个设计提供了一个比较全面的框架。我们可以从一个简化版本开始（例如只考虑 `S_interaction`, `S_source`, `S_recency`, `S_utility`），然后根据需要逐步引入更复杂的因素。你觉得哪些因素是最优先需要考虑的？

## 反思与整合

反思与整合（Reflection & Consolidation）过程的触发机制和频率对 `another-me` 的学习效率、成本控制和响应性至关重要。如果太频繁，会消耗大量计算资源（特别是 LLM 调用），可能还会在数据不足的情况下进行无效反思；如果太少，Agent 的学习就会滞后，无法及时捕捉用户的变化或形成深刻理解。

我们需要一个平衡且智能的触发策略。结合 `another-me` 持续运行、模仿学习的特性，目前来看一个**混合多种触发机制、动态调整频率和范围**的方案是比较理想的。

**核心触发机制:**

1.  **基于时间的周期性触发 (Scheduled Periodic Trigger):**
    *   **机制:** 按固定的时间间隔执行。
    *   **例子:**
        *   **每日深夜/低谷期:** 执行一次全面的、深度的全局反思，分析过去 24 小时甚至更长时间的记忆，寻找长期模式、总结主要活动、更新置信度较低或需要确认的 `UserPreference`。这是捕捉慢变量（如逐渐形成的习惯、兴趣领域的变化）的主要方式。
        *   **每小时 (可选):** 执行一次较轻量级的反思，主要关注过去一小时内未被实时事件触发反思的记忆，进行初步的模式识别或摘要。
    *   **优点:** 保证了规律性的学习和整理，避免遗漏。
    *   **缺点:** 可能在无事发生的时间段进行无效反思，无法快速响应突发事件。

2.  **基于关键事件的实时/近实时触发 (Key Event Trigger):**
    *   **机制:** 当发生特定类型的、具有高信息价值的事件时，立即或尽快触发一次**聚焦型**的反思。
    *   **例子:**
        *   **收到明确的用户反馈:** 用户直接评价 Agent 的行为（“做得好”、“不对，我应该会...”）、修正信息、或者给出明确指令偏好。这是最高优先级的触发器，需要立即分析反馈内容及其上下文，更新相关的 `MemoryUnit` 重要性，并创建/修改 `UserPreference`。
        *   **检测到显著矛盾:** Agent 的某个行为或推断 (`AGENT_THOUGHT`) 与一个高置信度的 `UserPreference` 发生了冲突。需要立即反思冲突原因，判断是偏好过时、情境特殊还是推断错误。
        *   **完成一个明确的任务/目标:** 如果 Agent 被赋予或自行设定了一个短期目标（如“查找关于 XX 的最新消息并总结”），在目标完成后进行一次针对该任务过程的反思总结。
        *   **遇到高度新颖/意外的情况:** `S_novelty` 分数极高的 `MemoryUnit` 出现，可能预示着用户行为或环境的重大变化，值得立即关注和分析。
    *   **优点:** 能够快速学习和适应关键信息，响应及时。
    *   **缺点:** 如果关键事件过于频繁，可能导致反思碎片化。

3.  **基于状态或阈值的机会性触发 (Opportunistic Trigger):**
    *   **机制:** 当系统满足某些条件时，利用空闲资源进行反思。
    *   **例子:**
        *   **Agent 空闲状态:** 检测到 Agent 一段时间内没有外部交互或内部活动（例如，用户离线，没有计划任务），可以利用这段时间进行反思，处理积压的记忆。
        *   **新记忆累积阈值:** 当未被反思的 `MemoryUnit` 数量达到一定阈值（例如 100 条）时，触发一次批处理反思，对这些新记忆进行整合。
        *   **低系统负载:** 检测到服务器 CPU/GPU 负载较低时，可以启动反思进程。
    *   **优点:** 有效利用资源，避免在高峰期增加负担。
    *   **缺点:** 反思的时机可能不确定。

4.  **用户主动触发 (User-Initiated Trigger):**
    *   **机制:** 用户通过界面或指令明确要求 Agent 进行反思或解释其行为/偏好。
    *   **例子:** “复盘一下我昨天下午浏览的内容”，“你为什么认为我对这个话题感兴趣？”
    *   **优点:** 提供了用户干预和理解 Agent 内部状态的途径。

**频率和范围的动态调整:**

*   **初始学习阶段:** 在 Agent 刚开始学习用户时，可以适当提高**周期性触发**的频率（例如每天多次），并对**事件触发**更敏感，以加速模型的建立。
*   **成熟稳定阶段:** 当 Agent 对用户的理解比较深入且用户行为模式相对稳定时，可以降低周期性触发的频率，更依赖于关键事件触发和机会性触发。
*   **反思范围:**
    *   **聚焦型反思 (Focused Reflection):** 由关键事件触发，只分析与该事件直接相关的短期记忆序列和关联实体/偏好。速度快，针对性强。
    *   **主题型反思 (Thematic Reflection):** 由任务完成或新颖主题触发，检索并分析与该主题相关的所有 `MemoryUnit` 和 `Entity`。
    *   **全局型反思 (Global Reflection):** 由周期性触发（如每日）或用户请求触发，分析较大时间范围或所有记忆，寻找跨领域的模式和关联。计算量最大。

**综合考虑:**

可以采用一个**混合策略**：

1.  **设定每日深夜的全局反思**作为基线。
2.  **实现对高优先级关键事件（用户反馈、显著矛盾）的近实时聚焦型反思**。
3.  **利用 Agent 空闲时间和/或新记忆累积阈值进行机会性的主题型或批处理反思**。
4.  **提供用户主动触发全局或主题反思的接口**。
5.  **考虑在 Agent 的不同生命周期阶段调整反思的频率和默认范围**。

这样可以在保证学习效率和响应速度的同时，也考虑到资源消耗和系统负载。当然，具体的频率、阈值和范围都需要在实际运行中不断调整优化。

## 处理记忆冲突

处理记忆冲突是确保 `another-me` 能够准确、一致地模仿用户，并适应用户变化的关键环节。冲突不是坏事，它们是学习和修正的绝佳机会。

以下是处理记忆冲突的一些策略和流程设计：

**1. 冲突的类型:**

首先，我们需要识别可能出现的冲突类型：

*   **新旧信息冲突 (`MemoryUnit` vs `MemoryUnit`):** 新观察到的信息与数据库中旧的、但仍然被认为是相关的记忆相矛盾。例如，用户过去常去 A 网站，但最近频繁访问 B 网站获取同类信息。
*   **信息与推断冲突 (`MemoryUnit` vs `UserPreference`):** 新观察到的行为或用户的直接反馈与 Agent 之前推断出的用户偏好/习惯相矛盾。例如，Agent 推断用户不喜欢某类新闻，但用户今天却主动搜索并阅读了这类新闻。
*   **推断间冲突 (`UserPreference` vs `UserPreference`):** Agent 推断出的两个用户偏好似乎相互矛盾。例如，既推断出“用户注重隐私保护”，又推断出“用户乐于在社交媒体分享生活细节”。这可能意味着需要更细致的上下文区分。
*   **行为与推断冲突 (`ACTION`/`AGENT_THOUGHT` vs `UserPreference`):** Agent 自己的行为或思考过程与它所理解的用户偏好发生冲突（这通常在规划或反思阶段被发现）。

**2. 冲突检测机制:**

冲突主要在以下环节被检测到：

*   **反思与整合阶段:** 当 Agent 分析一批 `MemoryUnit` 以生成或更新 `UserPreference` 时，会主动检查新信息与现有推断的一致性。
*   **记忆检索阶段:** 当为特定任务检索记忆时，可能会同时检索到语义上或逻辑上相互矛盾的信息。
*   **LLM 处理上下文时:** LLM 本身在处理包含冲突信息的上下文时，可能会在其推理过程中识别出不一致性（如果 Prompt 设计得当）。

**3. 冲突处理策略与流程:**

一旦检测到冲突，可以按以下步骤处理：

*   **Step 1: 识别与表征冲突**
    *   明确冲突双方是谁（哪个 `MemoryUnit`，哪个 `UserPreference`）。
    *   提取双方的关键属性：`timestamp`, `importance`, `source`, `confidence` (对于 `UserPreference`)，以及具体的 `content_text` 或描述。

*   **Step 2: 初步评估与优先级排序**
    *   **时间优先原则:** 通常，更新的信息更能反映当前状态。检查 `timestamp`，近期信息权重更高。
    *   **重要性/置信度原则:** `importance` 或 `confidence` 更高的一方更可信。
    *   **来源可靠性原则:** 用户直接输入 (`USER_INPUT`, `USER_FEEDBACK`) 的可靠性最高，其次是 Agent 基于大量证据的推断 (`UserPreference` 且 `confidence` 高)，再次是单次观察 (`OBSERVATION`)。
    *   **明确性原则:** 明确的陈述（如用户直接说“我不喜欢 X”）通常比模糊的观察（用户跳过了 X）更强。

*   **Step 3: 寻求用户澄清 (如果可能且必要)**
    *   **时机:** 当冲突显著（涉及高重要性/置信度项）、Agent 无法基于现有信息明确判断、或需要确认用户偏好是否发生改变时。
    *   **方式:** 设计友好的交互方式询问用户。例如：
        *   "我注意到您最近经常访问 B 网站，而不是以前常用的 A 网站了。您是对 B 网站更感兴趣了吗？"
        *   "我之前记录您可能不太关注 X 类新闻，但今天看到您在阅读它。是您改变想法了，还是有什么特殊原因呢？"
        *   "我发现您既注重隐私，有时也会分享生活细节。您是如何平衡这两点的呢？在什么情况下您会倾向于分享？"
    *   **价值:** 用户的直接反馈是解决冲突和修正模型的最可靠途径。

*   **Step 4: Agent 自主推理与调和 (LLM 介入)**
    *   **时机:** 当无法立即获得用户反馈，或冲突相对轻微时，或者作为用户反馈后的补充处理。
    *   **输入:** 将冲突双方的信息及其元数据提供给 LLM。
    *   **任务 (Prompt Engineering):**
        *   要求 LLM 分析冲突的性质和可能的原因（是信息过时？情境不同？推断错误？还是需要更细致的理解？）。
        *   要求 LLM 基于优先级原则和上下文提出解决方案：
            *   **更新:** 明确哪一方的信息更可信，建议降低另一方的 `importance` 或 `confidence`，或者标记为“可能已过时”。
            *   **修正:** 修改 `UserPreference` 的描述，使其更精确或包含例外情况（例如，“用户通常注重隐私，但在与亲密好友的小范围社交圈内乐于分享生活细节”）。
            *   **保留并标注:** 如果冲突代表了用户内在的复杂性或矛盾性，LLM 可以建议保留双方，并添加注释说明这种张力或需要注意的上下文。
            *   **寻求更多信息:** 如果信息不足以判断，LLM 可以建议 Agent 在未来关注哪些方面以收集更多证据。

*   **Step 5: 更新记忆系统**
    *   根据用户反馈或 LLM 的推理结果，执行更新操作：
        *   调整 `MemoryUnit` 的 `importance`。
        *   调整 `UserPreference` 的 `confidence` 和 `description`。
        *   可以创建新的 `MemoryUnit` (type=`CONFLICT_RESOLUTION`) 来记录这次冲突及其处理结果，为未来的元认知提供依据。
        *   在图数据库中，可以添加或修改关系，例如标记一个 `MemoryUnit` 或 `UserPreference` 被另一个“取代”(`SUPERSEDED_BY`) 或“修正”(`REVISED_BY`)。

**关键原则:**

*   **冲突是常态:** 接受用户行为和偏好的动态性和复杂性，将冲突视为学习信号。
*   **用户是权威:** 尽可能利用用户反馈来解决核心冲突。
*   **保留上下文:** 即使某个信息被认为是“过时的”，也尽量不物理删除，而是通过降低重要性或添加标记来处理，因为历史信息可能在特定情境下仍有价值。
*   **记录过程:** 记录冲突处理的过程本身也是一种有价值的元记忆。

通过这套机制，`another-me` 可以在面对不一致信息时，既能保持模型的准确性，又能不断深化对用户的理解，而不是变得混乱或固执。
